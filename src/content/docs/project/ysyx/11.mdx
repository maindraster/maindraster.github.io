这两篇文章的核心在于设身处地，如果你不希望被别人怎样的提问，你就应该怎样去提问，那么这么看来如何聪明地提问就显而易见。

## 别人帮你不是本分

> 向帮助你的人付费是一个高尚的行为。即使回答你的人不是为了钱。

他们没有义务一定要帮你，因此你提问想得到答案时你是“孙子”，他们是“大爷”，你的语言表达种种都应该方便他们、让他们感到舒适。

当然，提问之前应该先去自己看手册、善用搜索，比如明确不同系统之间的差异，这样不至于提出愚蠢的问题。

## 方便回答者

好的提问应该十分直接明确，因此它不能有太多无关和主观的废话。

### 关于提问

**问题的构成**

1. 我的目标是什么
2. 我使用的工具、环境等是怎样的
3. 我在达成目标的过程中做了什么
4. 最终产生了什么与预期不符的结果
5. 明确表述需要回答者做什么

**显然无关和主观的废话就包括**：
- 给出自己的猜测
- 无意义低声下气的表述
问题的形态
问题不在长，而在精。

精一方面强调尽可能短。因此方便回答问题的人就应该用尽可能少的语言描述得尽可能全面。例如，问题报告不应该全部截取，而应该截取其中关键的部分，减少对方的阅读负担。

精另一方面强调问题应该具体。文章中给出了很多对比的例子来说明具体。例如：问`我想更好地理解 X，可否指点一下哪有好一点的说明？`通常比问`你能解释一下 X 吗？`更好。

### 关于提问途径

当某个项目提供开发者邮件列表时，要向列表而不是其中的个别成员提问，即使你确信他能最好地回答你的问题。

在合适的地方提问：不要在与主题不合的论坛上贴出你的问题，也不要在探讨进阶技术问题的论坛张贴非常初级的问题。选择网站或论坛要注意如果某个问题只在特定环境或工具的版本发生就应该先去这个环境或工具的论坛提问，再去该项目的网站或论坛提问。

## 让回答者舒适

方便回答者重点在提问的内容和途径，而让回答者舒适重点在问题的呈现。

### 关于问题的形态

在文章中提到要使用易于读取且标准的文件格式发送问题，要注意：
- 编码的问题
- 使用纯文字而不是 HTML
- 不要发送一段文字只是一行句子但自动换行后会变成多行的邮件
- 不要指望黑客们阅读使用封闭格式编写的文档，像微软公司的 Word 或 Excel 文件等
- 不要使用表情符号

### 提问的态度

要诚恳有礼貌，**提问之后就说谢谢而不是得到回答之后感谢**。多用请和谢谢您的关注，或谢谢你的关照。让大家都知道你对他们花时间免费提供帮助心存感激。

即使你很急也不要在标题写紧急，这可能让回答者们反感。

## 其他

文章说技术大佬们喜欢直截了当的交流，因此可能存在看似“冒犯”的表述，这可能让人不适。我前面归纳的“舒适”重点在于从对方角度看，而不是从其他人的角度来感受。

如果还是搞不懂，比如对方的回答比较粗略，别立刻要求对方解释。要像以前试着自己解决问题时那样（利用手册，FAQ，网络，身边的高手），先试着去搞懂他的回应。如果你真的需要对方解释，要表现出我从中学习到的东西。
